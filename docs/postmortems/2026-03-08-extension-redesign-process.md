# Extension 重新设计 — 设计流程复盘

**日期**：2026-03-08
**类型**：设计流程复盘
**关联规范**：`docs/specifications/feature-extension-redesign.md`

## 事件

mutbot Session Channel 架构重构中，需要在 Extension 上声明可变默认值字段（`_channels: list[Channel]`），触发了 Extension `field()` 支持的设计需求。在设计 `field()` 支持之前，先与用户讨论 Extension 的整体设计方向，发现 Extension 存在三个结构性问题（无注册机制、操作语义混淆、字段声明不结构化），最终从"field 支持"扩展为"Extension 完整重新设计"。

整个设计过程产出：
- 架构设计文档 `docs/design/architecture.md`
- 功能规范 `docs/specifications/feature-extension-redesign.md`（9 个设计决策、5 阶段 14 个 Task、风险审阅）

## 方法论

### 从具体问题出发，但不急于解决

起点是 `field()` 支持这个具体问题，但没有直接跳进实现。用户主导了"先讨论设计方向"的决定，发现了比 `field()` 更根本的注册、枚举、操作语义问题。

**教训**：具体问题往往是设计缺陷的症状。解决症状前先诊断根因，可能发现需要的是重新设计而非打补丁。

### 借鉴但不照搬外部系统

参考了 Unity Component、Unity ECS (DOTS)、UE5 Mass 三个系统，提取了共性模式（注册、枚举、生命周期、组合查询），然后按 mutobj 的定位逐个判断取舍。最终借鉴了组件注册和枚举，但明确不采用纯 ID 实体和强制 Archetype 声明。

**教训**：外部系统是思维工具，不是设计模板。关键是提取正交的设计轴，在每条轴上独立做决策。

### 用正交轴分解张力

多个设计目标之间的矛盾（可扩展 vs 可优化、OOP vs DOD、显式 vs 简短）被拆解为四条正交轴（数据模型、组合方式、操作语义、优化边界），在每个维度上独立收敛。

**教训**：设计张力不是"二选一"，而是在不同维度上可以各自取最优值。

### 区分"概念"和"实现细节"

Archetype 的讨论是关键转折点。用户问"Archetype 是实现细节还是概念设计？"，直接决定了设计方向——如果是概念，扩展性受损（新增 Extension = 修改 Archetype）；如果是实现细节，两者不冲突。

**教训**：对每个设计元素都要问"这是用户需要理解的概念，还是运行时的实现细节？"。概念越少，心智负担越低；实现细节越多，优化空间越大。

### 命名是设计决策，不是风格偏好

`of` → `get_or_create` / `get` 的命名讨论通过系统分析 Python 生态的命名先例（`dict.get`、`dict.setdefault`、Django `get_or_create`）、考虑 AI 生成代码时的行为、评估误用的后果方向来选择，最终选择了"不可能理解错"的命名。

**教训**：好的命名让错误用法在视觉上就不对。命名应该通过分析误用场景来选择，而非凭感觉。

### 设计决策应消除风险，而非缓解风险

风险审阅中两个"中等风险"通过设计变更直接消除：
- 中间类识别 → 改用 `__init_subclass__` + `__orig_bases__`（标准 Python 泛型模式），中间类不存在了
- Extension 继承注册歧义 → "有 `Extension[T]` 就注册"，规则无歧义了

**教训**：如果风险需要复杂的缓解方案（标记、检测、特殊处理），先考虑能否改变设计让风险不存在。好的设计不是"处理了所有边界情况"，而是"边界情况本身不存在"。

### 符合直觉是核心约束

Extension 继承注册的最终规则（"有 `Extension[T]` 就注册"）之所以好，不是因为它技术上最优，而是因为它符合"你写了什么就意味着什么"的 Python 哲学。过程中曾考虑 `register=True/False` 参数，但最终因为"Extension[T] 语法本身就是显式的注册声明，额外参数是多余的"而放弃。

**教训**："符合直觉"不是可选的加分项，而是硬约束。设计完成后要回答："一个不读文档的开发者，看到这段代码会怎么理解？"如果理解错了，设计有问题。

## 工作规范

### SDD 流程

- **先讨论后更新文档**：对于方向不明确的设计，在对话中自由讨论比每轮修改文档高效。文档记录已确认的决策，而不是讨论的中间产物
- **文档范围可以扩大**：起始 `feature-extension-field.md` 最终重命名为 `feature-extension-redesign.md`。当发现问题范围大于初始认知时，应主动扩大规范范围
- **设计文档与架构文档分离**：架构理念（`docs/design/`）记录长期有效的设计原则，功能规范（`docs/specifications/`）记录具体功能的决策和实施步骤
- **决策记录推导依据**：每个决策不仅记录结论，还记录"为什么"和排除了什么。这让未来的维护者（包括 AI）理解决策背后的权衡

### 人机协作

- **用户提供约束和直觉，AI 提供分析和选项**：用户说"参考 ECS 但不要照搬"，AI 做系统对比分析；用户说"符合直觉最重要"，AI 用这个标尺筛选方案
- **用户追问推动深入**："Archetype 是实现细节还是概念设计？"这类追问迫使分析从表面深入到本质
- **AI 应主动暴露假设**：最初假设"类级注册为主（archetype）"，用户追问后推翻。如果 AI 没有把假设摆出来，用户就没有机会挑战它
- **AI 应明确表达自己的判断**：用户问"对 AI 来说最符合直觉的是什么？"时，不应回避。诚实分析 AI 视角下的优劣，给出明确判断
- **讨论中不怕推翻前论**：`register=True` 默认值 → 考虑 Declaration 继承场景后 → 改回无参数方案。好的设计过程中推翻前论是正常的

### 风险审阅

- **风险审阅在实施步骤之后**：先规划"怎么做"，再审视"可能出什么问题"
- **设计风险和实现风险分开处理**：设计风险需要回到设计阶段解决（可能改变 API）；实现风险在实施中处理即可
- **用"这是标准 Python 做法吗？"检验设计**：中间类问题正是通过这个问题暴露的——`__getitem__` 创建真实类不是标准泛型行为，因此是设计问题而非实现问题

## 洞察

### AI 友好 = 人类友好

在多个决策点用"AI 会怎么理解/生成"作为评判标准，最终发现 AI 友好的设计同时也是人类友好的。两者都受益于：自文档化的接口（方法名就是行为描述）、结构化的元信息（类型签名而非 duck typing）、显式优于隐式（`Extension[T]` 语法即注册声明）。

通用启发式：**如果 AI 可能用错，新手也可能用错。**

### "概念不对称，基础设施对称"

当系统中有两种表面相似但概念不同的东西（Declaration 字段 vs Extension 字段），概念层保持它们的区别（公开接口 vs 私有扩展状态），但基础设施层统一处理（都用 `field()`、都可内省）。这样既保持了概念清晰，又避免了基础设施的重复建设。

### 渐进式优化需要"留口但不暴露"

"阶段 1 的 API 写法不能和阶段 N 的优化冲突"——要求设计初期就为未来优化留出空间（结构化 field 声明、注册表、可内省 schema），同时不暴露优化细节给用户（Archetype 是实现细节）。

## 流程模式

```
具体问题 → 根因分析 → 方向讨论（不碰细节）
    → 借鉴外部系统 → 正交轴分解 → 逐轴收敛
    → API 设计 → 命名推敲 → 实施步骤
    → 风险审阅 → 设计变更消除风险 → 最终确认
```

关键原则：**先收敛方向，再展开细节**。在方向明确之前不讨论 API；在 API 确定之前不讨论实现；在实现规划之后才审阅风险；风险可能触发设计回溯。
